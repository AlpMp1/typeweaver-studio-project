我正在构建 TypeWeaver Studio，一个面向历史 JavaScript 项目的 AI 类型迁移与代码质量修复工作台，目标是帮助个人开发者和小团队把长期迭代的 JS 项目安全、渐进地迁移到 TypeScript。很多项目早期为了快速交付没有建立完整类型体系，随着功能增加，API 响应结构、组件 props、service 参数、工具函数返回值和异常分支会逐渐变得隐式化。开发者在修复 bug、升级框架或重构模块时，经常需要反复阅读代码、运行测试、查看错误栈和历史提交，才能判断某个字段到底是否可空、金额到底是字符串还是数字、组件是否依赖某个隐藏属性。这类问题非常适合用长上下文和多轮 AI Coding 工作流解决。

TypeWeaver Studio 的核心流程由多个 Agent 协作完成。Repository Architect Agent 会读取项目目录、关键源码、package 配置、API 样本、测试快照、运行时错误日志和历史 bugfix 提交，生成模块边界、调用链和高风险隐式类型清单。Type Inference Agent 会从真实使用场景中推断 DTO、组件 props、service 参数和工具函数返回值，识别 any、unknown、nullable、字符串数字混用等风险点。Migration Planner Agent 会把迁移任务拆成小步，例如优先迁移 API 边界、订单实体、用户实体、价格规则或组件 props，并给出影响范围、验收标准和回滚方案。Coding Agent 会在 OpenClaw、OpenCode 等 AI 编程工具中调用 MiMo 生成 TypeScript 类型定义、修改源码、补充回归测试，并根据 typecheck、lint、unit test 和 snapshot diff 输出进行多轮修复。Review Agent 最后会输出 PR 级报告，包括修改摘要、行为一致性检查、受影响模块、风险等级和下一步建议。

这个项目非常依赖较高的 Token 额度。单个仓库的类型迁移不是局部问答，需要同时理解源码、测试、接口样本、错误栈和历史修改；单个迁移单元通常包含“仓库理解—类型推断—迁移规划—代码生成—类型检查—测试失败分析—补丁修复—风险审查—报告生成”等多个回合。我计划使用 20 到 30 个真实开源 JavaScript 项目做 benchmark，覆盖 React、Next.js、Node.js 服务、工具函数库等不同技术栈，每个项目选择 5 到 10 个真实迁移任务，评估 MiMo 在长上下文代码理解、类型推断、渐进迁移、测试补齐和 PR Review 中的效果。由于需要大量读取仓库上下文并进行多轮实验，希望申请较高 Token Plan 额度，用于持续完成 demo、评测和迭代。
